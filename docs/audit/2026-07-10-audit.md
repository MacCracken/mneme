# mneme — Security Audit (2026-07-10, v1.0.0)

**Scope:** the Cyrius port (`src/*.cyr`), reviewed for vulnerabilities the port
could introduce relative to the Rust original (`rust-old/`, the parity oracle).
**Reviewer:** port author, pre-1.0.0 release. **Method:** manual review of every
subsystem's untrusted-input paths + memory handling, cross-checked against the Rust
behavior. **Result:** no port-introduced high/critical findings. Two inherited
design considerations are documented below.

## Threat model

mneme is a local, single-user knowledge base + TUI + in-process HTTP API. Untrusted
input surfaces: note titles/content/tags, HTTP request bodies (api), MCP tool args,
clipped HTML, and daimon HTTP responses. It is **not** a multi-tenant network service
— the `mneme-api` router runs in-process (no listening socket; `tower::oneshot`
parity), so there is no unauthenticated remote attack surface in the shipped binary.

## Findings by area

### 1. SQL injection (patra) — MITIGATED
`store_db` builds SQL by string concatenation, but every string value goes through
`_db_sql_str`, which wraps in single quotes and escapes embedded quotes (`'` → `''`).
UUIDs/ints are formatted numerically. No untrusted value reaches SQL unescaped.
Reviewed: insert/get/update/delete note + tag, joins, list. **No injection.**

### 2. Path traversal (file store) — MITIGATED (title path); INHERITED (explicit path)
`title_to_path` slugifies (alnum→lowercase, all else→`-`, runs collapsed), so a
hostile title (`../../etc/passwd`) sanitizes to a flat slug — no traversal. **Note
(inherited from Rust):** `CreateNote.path` is an *optional explicit* path written
as-given (`store_vault.create_note` → `filestore_write_note`), matching the Rust
`path: Option<String>`. A caller supplying `../x` could write outside the vault.
This is **parity with Rust, not a port regression**; callers (api/mcp) should treat
the `path` field as trusted or add a `..`/absolute-path reject. Filed as a follow-up.

### 3. Memory safety (Cyrius manual model) — REVIEWED
The port uses raw `alloc`/`load64`/`store64` over an alloc-once arena (no free →
no use-after-free / double-free). Risks are struct-offset/bounds errors; mitigated
by the 61-file test suite exercising every accessor. All hand-written parsers loop
under an explicit length (`while i < n`) with no NUL-termination assumptions —
reviewed the flat-JSON getters (api/mcp/embedding), tokenizers, and markdown/HTML
scanners for over-read: none found. Fixed 1 MB read buffers (`file_read_all(_,
1048576)`) truncate oversized notes (availability, not memory corruption). The
"13 MB static data" note is `sigil`'s crypto tables — benign.

### 4. Untrusted input parsing (JSON / HTML / query) — REVIEWED
Request-body and tool-arg parsing are bounds-checked hand-parsers; output JSON is
escaped (`_api_esc`/`_net_esc` handle `"`,`\`,control chars). `mcp_parse_uuid`
validates a 32-hex-digit shape before parsing. HTML clipping (`io_web_clipper`)
strips tags with bounded scans. No format-string or eval surface exists.

### 5. Network / TLS / secrets — REVIEWED
Outbound HTTP (daimon, remote embeddings) uses `sandhi`; `https://` URLs get
`tls_native` (cert-window checked against the RTC on AGNOS). The `api_key` is sent
as a `Bearer` header over that channel and is never logged. On failure the bridges
degrade to `0`/local — no secret or partial state leaks. There is no inbound socket
in the binary. Timeouts/blocking are bounded by sandhi's connect/recv deadlines.

### 6. Randomness — REVIEWED
UUID v4 uses `getrandom` (real kernel entropy on AGNOS/Linux, never a zeroed
buffer); v5 uses SHA-1; content hashing uses `sigil` SHA-256. No weak PRNG in a
security-relevant path.

## Conclusion

**Pass for v1.0.0.** No port-introduced high/critical vulnerabilities. One inherited
consideration (explicit `CreateNote.path` traversal, identical to Rust) is documented
as a hardening follow-up. Re-audit when: a listening socket is added, local ONNX
weights are loaded from disk, or the `path` field is exposed to untrusted callers.
